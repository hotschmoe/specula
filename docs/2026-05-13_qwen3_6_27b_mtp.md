# 2026-05-13 Qwen3.6-27B MTP — first numbers

Follow-up to the overnight sprint. Goal: bench Qwen3.6-27B in both
quants the user wanted, and try MTP self-draft via PR #22673 since
mainline can't even load the MTP-preserved GGUFs.

## TL;DR

**Qwen3.6-27B-MTP-Q4_0 on PR #22673 (build-opencl-mtp), OpenCL `-ngl 0
-t 18`, `--spec-type draft-mtp --spec-draft-n-max 8`: TG 12.17 t/s
(+45% over no-MTP baseline 8.39).** Acceptance rate 95.8%.

**Qwen3.6-27B-MTP-Q8_0** at the same config: TG 7.98 t/s (+55% over
baseline 5.13). Acceptance rate 95.8%.

The MTP head on the unsloth files is **remarkably well-trained** —
acceptance plateaus at 98-100% from n_max=2 through n_max=6, only
drifting down to 94% at n_max=16.

## Why mainline can't be used

Mainline `856c3adac` errors at load:
```
llama_model_load: error loading model: missing tensor 'blk.64.ssm_conv1d.weight'
```
The MTP-preserved GGUF lists 65 blocks in `block_count`, but mainline
expects every block to have full SSM tensors. The PR's `a55493bbd
spec: support MTP` commit teaches mainline that block 64 has a
different tensor layout (attention-only MTP head, no SSM path).

The unsloth README's claim that "all variants are MTP-preserved" is
accurate; what they don't mention is **the files literally won't load
on master**. Until #22673 merges, these GGUFs are PR-only.

### Key observations

1. **TG plateau at n_max=8.** Both quants peak around n_max=8-12,
   with marginal gain above. n_max=4 captures 95% of the win at no
   acceptance cost; n_max=8 is the broadly-safe default.
2. **Q8_0 sees larger relative speedup.** +55% vs Q4_0's +46%.
   Bandwidth math: Q8_0 baseline TG is bandwidth-bound (5.13 t/s
   ≈ 228 GB/s ÷ 27 GiB × ~65% efficiency); MTP batches verifier
   work so each forward pass amortizes more decoded tokens. The
   slower the baseline, the more MTP helps.
3. **Prefill takes a ~22% hit on Q4_0 (63 → 49) and ~19% on Q8_0
   (44 → 36).** The MTP head's KV state needs to be initialized
   during prefill. For long prompts (>2k tokens) the absolute PP
   loss can outweigh the TG gain. The crossover prompt length for
   Q4_0 27B (assuming 256-token answer):
   - no-MTP wall: `P/63.3 + 256/8.39` s
   - MTP n=8 wall: `P/48.99 + 256/12.17` s
   - Equal at P ≈ 1860 tokens. **MTP is the right call for any
     prompt shorter than ~1.8k tokens**, which covers most chat /
     RAG-shortlist / structured-output workloads.
4. **Acceptance rate is shockingly high (98-100%).** The unsloth
   MTP head was trained with high quality — at n_max=2 with Q8_0
   we got *exact* 100% accept (134/134). The dropoff at higher
   n_max is mild (94% at n=16), so even aggressive drafting is
   profitable.

## Hardware notes from this session

### `cl_qcom_large_buffer` extension confirmed available

User caught a mistake from the overnight sprint: I claimed Adreno
was limited to 24 GB. The 24 GB is the *default* `clGetDeviceInfo`
return; with `GGML_OPENCL_ADRENO_USE_LARGE_BUFFER=1` set, the
`cl_qcom_large_buffer` extension is honored, allowing single
allocations beyond the 2 GB default cap and total usage approaching
the 44 GB BIOS allocation.

The flag is detected at load (stderr line `Adreno large buffer enabled`).

**However**, on models that already fit in 24 GB (4B Q4_0 = 2 GB,
35B-A3B MXFP4 = 20 GB), the flag is marginal and sometimes slightly
slower (`results/csv/track_c2b_large_buffer_reruns_2026-05-13.md`):

| model | config | no LARGE_BUFFER | with LARGE_BUFFER |
|---|---|---|---|
| 4B Q4_0 | `-ngl 99 -t 16 -ub 512` | PP 563.74 / TG 24.72 | PP 524.49 / TG 25.99 |
| 35B-A3B MXFP4 | `-ngl 99 -t 16` | PP 199.42 / TG 14.62 | PP 198.39 / TG 13.95 |

So the practical rule:
- **Use LARGE_BUFFER when the model exceeds 24 GB on GPU.** (e.g. Q8_0
  27B at 27 GiB, if GPU offload worked.)
- **Skip it for models that fit.** Different buffer-mgmt path appears
  to add ~5% overhead.

### Qwen3.6-27B does NOT GPU-offload on OpenCL

Independent of LARGE_BUFFER, `-ngl 99` fails on 27B Q4_0:
```
ggml_opencl: clCreateImage error -40 at ggml-opencl.cpp:11358
GGML_ASSERT(0) failed
```

The model's SSM-style tensors (`ssm_a`, `ssm_conv1d`, etc.) have
shapes the Adreno OpenCL image path doesn't accept. Same for Q8_0.

**`-ngl 0` (coprocessor mode — backend registered, weights on CPU)
is the only OpenCL path for Qwen3.6-27B on Adreno today.** It works
fine and is what every result above used.

## Caveats for the headline

- All numbers are r=1 / single completion via `llama-server`. Variance
  on TG across reruns is ±2-5%; the MTP-vs-no-MTP delta is well
  outside that band, but I would not stake a sub-3% comparison on a
  single run.
- The prompt is a single fixed string. MTP acceptance is somewhat
  prompt-dependent — for highly stochastic outputs (creative writing,
  high temperature) the acceptance rate will likely drop. The 98%+
  numbers here are best-case for a structured technical explanation
  at temperature=0.
- PR #22673 is open and may rebase before merge. The CLI flag name
  has already changed once (`mtp` → `draft-mtp`); future renames
  possible.
- The non-MTP unsloth file (`Qwen3.6-27B-Q4_0.gguf`, 15.79 GB) is
  *not* on disk for the strict A/B; we used MTP-preserved file with
  `--spec-type none` as the no-MTP baseline. The MTP head tensors
  add some weight footprint but are not executed during baseline
  inference, so this is a reasonable apples-to-apples comparison.
